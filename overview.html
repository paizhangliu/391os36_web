File System

The 391OS-36 File System has a total size of 8MB, and is divided into 4KB blocks of one boot block, several inodes, and data blocks.
Each boot block can track up to 62 inodes and one root directory, and each inode can track up to 1023 data blocks.
Thus, the limitations are up to 62 files of 4092KB size and 32 characters name length. Also, it does not support hierarchy and is read-only.

File System Abstraction

The 391OS-36 treat any files, device (RTC), and directory as files (see above). Each process has a dynamic File Descriptor (FD) that supports 8 open files.
Each FD entry tracks inode, position, flags, and an operation jump table, according to the file type. They are directly utilized by system calls (see below).

Context Switch/Assembly Linkages

Whenever an interruption, exception, or system call happened, the processor jumps to a specific assmebly linkage from the IDT.
This procedure is sometimes called a "context switch". It is almost constantly happening with the scheduler enabled, as the OS schedules the next program.
Assembly linkages for interruption, exception, and system calls are different, but they all construct a hardware context for furture restoring or usage.
They will also push/pass through arguments accordingly, and call the corresponding handlers (see below).
When returning back from the handler, assembly linkage will resume the hardware context saved before by IRET.
Or, halt/execute/process switch (see below) constructs a custom context, then it will jump to a new or background program's Prgram Counter (PC) and stack.
When going back to the user space, it will also dispatches any pending signal (see below, optional).
Now, you see the reason why context switch is expensive. The assembly linkage really needs to deal with a lot of stuff.

Exception/Exception Handlers

An exception happens when the user program did something wild, such as dividing by zero or accessing illegal pages (see below).
Exceptions are synchronised. When an exception was generated, the processor jumps to the specific assembly linkage (see ablve).
The exception assembly linkage pushes information and calls an unified exception handler.
Then, the handler will display the error message, debug information, and send a signal to the process for further handling (see below).
If you did not implement the signal or no handler was installed, the default behavior is usually killing the offending program.

Interrupt/Interrupt Handlers

An interrupt happens when a piece of hardware needs processor's attention, such as keyboard input, RTC, and PIC.
391OS-36 only contains essential drivers (see below). Interrupt lines (IRQs) from non-supported hardware are masked and ignored.
The IRQ lines are managed by two i8259 PICs just like any other classic IBM-compatible PC.
Interrupts are non-synchronised. When an interrupt is raised and got processor's attention, it jumps to the specific assembly linkage (see above).
The interrupt assmebly linkage pushes information and calls an unified interrupt handler.
Then, the handler will call the corresponding driver, and the driver will handle the remaining work (again, see below).

Type Generated By Asynchronous Unexpected
Interrupts External device needs attention YES YES
Exceptions User program behaving wildly NO YES
System Calls (see below) User program, deliberately via INT x80 NO NO

Supported Devices/Drivers

The 391OS-36 has drivers for the standard keyboard, Programmable Interval Timer (PIT), Real-Time Clock (RTC), and terminal.
Keyboard, PIT, and RTC drivers are for real devices and contains handlers for their interruptions (see above).
The standard keyboard driver is responsible servicing key presses, and it also has support for the capital letter and combinational key handling.
The PIT and RTC all generates interrupts periodically.
The PIT and PIT driver are used to handle the scheduling and can be turned off (see below). It is only accessible by the 391OS-36 kernel.
The RTC is user-level accessible and it is "virtualized" for each process by the RTC driver, so they all have its own instance and frenquency.
Now, you see why the OS should use PIT for its task scheduling. You don't want an user program to adjust the scheduler frequency in the kernel.
The terminal driver is for standard input and output and works as a bridge with each terminal, keyboard driver, and system calls.
There are three terminals (see below). The input buffer for each terminal is limited to 128-characters.

Memory Addressing

The 391OS-36 has bypassed segmentation just like any mainstream modern operation systems. Only paging was used for the memory addressing.
The memory layout 391OS-36 used is fixed (see below).

Physical Memory Layout

From low to high, the 391OS-36 has utilized the following 32MB physical memory:

  • 4MB space with five 4KB space for the VRAM for the active terminal, 3 terminal backups, and an extra backup space
  • 4MB space for the kernel data (mostly Process Control Blocks (PCBs) and Terminal Info (TI) blocks) and kernel stack
  • Six 4MB spaces for user applications' binary and stack (see below)

Virtual Memory Layout

From low to high, the 391OS-36 has mapped the 4GB virtual memory space for each user program as following:

  • 4MB managed by a superviser Page Table (PT), with five 4KB superviser pages, going through from the physical memory (see above)
  • 4MB superviser jumbo page for the kernel, going through from the physical memory (see above)
  • 4MB user page at address address 128MB, for the current running program only (see below)
  • 4MB managed by a user Page Table (PT), with a 4KB vidmap page for the current active program, controlled by the vidmap system call (see below)

System Calls

System Call Description
halt Halt the current program (process).
Free the PCB (see below), close the FD, tear down the vidmap and program page, and resume the parent program's PC and stack.
execute Load and execute a new program.
Sanity check, create the program page, allocate a PCB, create the FD, bookkeeping, change the stack and jump to the new PC.
read Read data from a opened file (see file system abstraction above).
write Write data to a file (support terminal and device/RTC only).
open Allocate a FD entry and open a file.
close Close the FD entry and close a file.
getargs Return the command line argument. It is extracted during the execute system call.
vidmap Giving the user-level access to the current VRAM by mapping it to the vidmap page, which is user-level accessible.
set_handler Set custom handler for a signal (see below).
sigreturn Restore the hardware context from the signal handler (see below).

Please note that the last two system calls are for extra credit; they are not required.
System calls are generated deliberately by user programs by generating a "soft interrupt," INT x80 on x86 systems.
When a system call was generated, the processor jumps to the system call assembly linkage (see above).
Arguments for system calls are carried by %EAX, %EBX, %ECX, and %EDX registers.
The linkage will check if the arguments are valid, and then hand the job to the corresponding system call handler.

Process Control

391OS-36 supports up to 6 processes. Each process holds a Process Control Block (PCB) in the kernel.
The kernel manages the PCB allocation by a custom design called PCB pool.
This custom design allows the 391OS-36 to allocate and free the PCB resources freely.
Each PCB tracks the allocated process ID (PID), parent PID, terminal ID, kernel/user stacks, FD, signals and its handlers, etc.
While 6 processes are supported, only 3 are actually multitasking, and 1 at each time is running/scheduled (see below).

Scheduler

391OS-36 supports a round-robin scheduler based on the PIT interrupt (see above) that can be turned off.
The scheduler schedules 3 processes that is shown on each terminal, called active processes, so they can multitask.
But only one program is "actually" running, called running process, as there are only one processor supported.
When switching the running program in the background, a lot of works were done. One of the big challenge is to do a switch with separate VRAM.
The switching begins with PIT driver, with the hardware context saved. It will then switch the page, stack, and Program Counter (PC) like a halt call.
Besides the bookkeeping and context switch, the background running process only prints to the VRAM of its terminal even it is not active.
For 391OS-36, this is done by having 5 individual VRAM pages .
Turning off scheduler will disable the multitasking, but it mimics the environment for your work prior to CP5.
In that case, program switching only happens when the user is switching the terminal in the foreground.

Multiterminals

391OS-36 supports up to 3 terminals. Each terminal holds a custom design called terminal info (TI) block in the kernel.
Each TI block tracks the PCB pointer, coordinations, RTC parameters, etc.
When switching, 391OS-36 will backup the current VRAM, copy the corresponding VRAM to the display, and remap the vidmap if necessary.

Signals

Please note that signals are for extra credit; they are not required.

Signal Sender Default Handler
DIV_ZERO Exception Handler Kill the process.
SEGFAULT Exception Handler Kill the process.
INTERRUPT Keyboard Driver Kill the process.
ALARM RTC Driver Ignored.
USER1 N/A Ignored.
SYSKILL Kernel Not user-installable, always kill the process.

The first five signals supports a custom user-installed handler through set_handler system call.
You will need to implement the first five to be eligible to receive extra credit for the signals in your MP assignment.
Linkage is carefully engineered to construct the stack, so signals are checked and delivered during any context switch (see above).
Signal requires a lot of bookkeeping including masks, handlers, pending signals in the PCB of each process.

Process Manager

The 391OS-36 Process Manager is a kernel utility that prints the current PCB Pool and TI block information:

  • PID, the Process ID
  • TID, the Terminal ID
  • PPID, the Parent Process ID
  • KSP, the Kernel Stack Pointer
  • FD, the File Descriptor
  • ECHO, the echo flag
  • VMAP, the vidmap flag
  • COOR, the cursor coordination
  • ... and also, the (active, for each terminal) program name
It is design specific to 391OS-36, so it is definiately not required.
Please note this utility's kill function does not work with the scheduler (please check other known issues in the demo page).
